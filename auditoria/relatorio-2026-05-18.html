🔒 Segurança

CRÍTICO

Chave secreta do banco usada em página pública

A página de orçamento público (/orcamento/[token]/page.tsx) usa a chave SERVICE_ROLE_KEY, que é uma chave de administrador do banco de dados. Embora ela só rode no servidor, se alguém conseguir injetar código nessa página, teria acesso total ao banco.

Impacto no negócio: Um invasor poderia ler, alterar ou apagar todos os dados dos seus clientes, pedidos e orçamentos.

✅ Correção: Criar uma função server-side (API route) que faz essa busca, e a página chama a API. Assim a chave nunca fica exposta nem indiretamente.
📄 src/app/orcamento/[token]/page.tsx:12-21
IMPORTANTE

Erros silenciosos — código engole exceções

Existem 6 pontos no código onde erros são capturados e ignorados (catch {}). Isso significa que se algo der errado (falha no banco, erro no PDF), você não vai saber — o app simplesmente para de funcionar sem avisar.

Impacto no negócio: Você pode perder dados salvos pelo cliente sem perceber. Um orçamento que não gerou PDF, um cliente que não foi salvo.

✅ Correção: Registrar o erro com console.error() e mostrar uma mensagem amigável ao usuário.

IMPORTANTE

Uso excessivo de "any" no TypeScript

Existem 20+ usos de any no código, principalmente na página de agendamentos. O any desativa a verificação de tipos do TypeScript — é como se você estivesse escrevendo JavaScript puro, perdendo toda a segurança que o TypeScript oferece.

Impacto no negócio: Erros que seriam detectados automaticamente pelo TypeScript passam despercebidos e só aparecem quando o cliente já está usando.

✅ Correção: Criar interfaces TypeScript para os dados de agendamento e documentos, substituindo todos os any.
📄 agendamentos/page.tsx (15+ usos) · clientes/[id]/page.tsx · dashboard/page.tsx

📁 Estrutura do Projeto

SUGESTÃO

Auth layout faz autenticação no client-side

O layout das páginas protegidas (src/app/(auth)/layout.tsx) é um client component que redireciona se não houver usuário. Isso significa que o conteúdo protegido é enviado ao navegador antes do redirecionamento.

Impacto: Leve — o middleware já protege antes, mas seria mais seguro fazer a verificação no servidor.

✅ Melhoria: Mover a verificação de auth para Server Components, usando redirect() do Next.js diretamente.

⚡ Performance

IMPORTANT

Consultas ao banco sem limite ou select específico

Várias consultas no dashboard e em outras páginas usam select('*') ou não limitam a quantidade de resultados. Isso traz dados desnecessários do banco, deixando o app mais lento.

Impacto no negócio: O app pode ficar lento conforme a quantidade de dados cresce. Clientes com muitos pedidos vão perceber lentidão.

✅ Correção: Especificar só as colunas necessárias (select('id, name, phone')) e adicionar paginação.
📄 dashboard/page.tsx · ClienteBusca.tsx · storage.ts
SUGESTÃO

Imagens sem otimização

As fotos dos pedidos são exibidas usando <img> em vez do <Image> do Next.js. O Next.js otimiza imagens automaticamente (redimensionamento, formato WebP, lazy loading).

Impacto: Fotos pesadas demoram para carregar, especialmente no celular.

✅ Melhoria: Usar next/image com loading="lazy" e tamanhos adequados.
📄 src/components/pedido/PhotoGallery.tsx
